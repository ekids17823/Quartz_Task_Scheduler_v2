# 企業級 Quartz 工作排程器 (Task Scheduler v2) - 使用者操作手冊

本手冊詳盡記錄了基於 Quartz.NET 與 C# WPF 打造的「企業級工作排程器」各項功能、系統邏輯及應用情境。本系統專為 24/7 全天候無人值守環境設計，並針對 Quartz 引擎常見的「補跑爆衝」與「時區鎖死」進行了深度客製化改良。

---

## 1. 系統架構與核心特色
* **核心引擎**：採用 `Quartz.NET` 提供極高併發且精準的排程調度。
* **資料庫架構**：採用 `SQLite` 並啟用 **WAL (Write-Ahead Logging)** 模式與 **記憶體暫存 (MEMORY Temp Store)**，徹底解決了高頻率、多工作業下常見的『Error 14: unable to open database file』鎖死錯誤。
* **分離式架構**：前端 UI (WPF) 與後端排程引擎 (ASP.NET Core API) 分離，確保介面關閉或當機，背景的排程任務依舊準時運行。
* **系統全自動清理**：內建防爆量清道夫，每天凌晨自動清理過期的排程紀錄與稽核紀錄，並執行 `VACUUM` 回收實體硬碟空間。

---

## 2. 新增與編輯排程 (Jobs)

在主畫面上點擊「建立工作」即可新增一項新任務。每個排程 (Job) 可掛載 **多個** 不同的觸發條件 (Triggers)。

### 2.1 基礎資訊
* **群組與名稱**：作為排程的唯一識別（例如：`DataBase` 群組下的 `DailyBackup` 任務）。
* **執行檔路徑**：您要系統定時執行的程式 (如 `.exe`, `.bat`, `.py`) 的完整路徑。
* **引數**：啟動該程式時所需要帶入的參數，如 `-mode silent -log trace`。
* **工作目錄**：該程式執行時所處的起始資料夾，這有助於腳本內相對路徑的讀取。

### 2.2 併發規則 (Concurrency Rule)
當同一個任務的「上一次執行」還沒跑完，但「下一次執行的時間」又到了，系統該如何處置？
1. **以平行方式執行新執行個體 (預設)**：不管上一次，時間到就啟動新的，兩者同時獨立並行。
2. **停止現有的執行個體**：強制中斷正在跑的舊版，並重新啟動一個全新的任務。(適用於長時間佔用但必須重置的腳本)。
3. **不要啟動新執行個體**：系統會安靜地略過這次觸發，直到上一班車跑完才允許開下一班。(適用於嚴格禁止重複操作的資料庫更新或備份)。

## 3. 觸發條件詳解 (Triggers)

本系統將任務條件分為四種常見類型，並針對 Quartz 底層的「預設馬上啟動」行為做了完美防堵。

### 3.1 僅一次 (One-Time)
* **邏輯**：只會執行一次，永遠不會重複。
* **防呆機制**：
  若您設定的起跑時間已經是「過去」，且您**沒有**勾選「錯過盡快執行」，這套系統會自動將該觸發條件攔下並丟棄，**絕對不會**像原生的 Quartz 一樣存檔後立刻「強制補跑」。雖然引擎接不到該指令，但這筆設定依然會完整保留在您的畫面上供未來編輯留底紀錄。

### 3.2 定期循環 (Interval)
* **邏輯**：從起跑點開始，每隔 X 分鐘/小時連續重複執行。
* **智慧延展機制 (Smart Start-Delay)**：
  在原生 Quartz 中，若您未特別指定未來時間，系統會將「起跑點」設為「存檔的當下」，導致您一按存檔就會立刻噴出第一筆執行紀錄（StartNow 行為）。  
  **本系統改良**：若您未勾選「錯過盡快執行」，系統發現您的起跑點是「現在或過去」，會自動替您精算出**未來的下一個週期點**，並將起跑線強制延展過去。
  *範例：您新建了一個每 5 分鐘執行的任務。存檔當下絕對不會馬上啟動，系統會讓它安靜地等待 5 分鐘後，才準時發動第一次觸發。*

### 3.3 每天 / 每週 (Daily / Weekly)
* **邏輯**：綁定特定星期或每天的特定時間執行。
* **機制**：系統會自動轉譯為進階 `Cron` 運算式處理，確保它能準時啟動且自動跨越午夜或跨日期的問題。

### 3.4 進階設定 (Cron Expression)
* **邏輯**：接受標準 6 段式 Cron 語法，例如 `0 0 12 L * ?` (每個月最後一天中午 12 點)。適用於極端複雜的條件要求。

## 4. 錯過排程的補發機制 (Misfire Action)

排程介面上有一項極度關鍵的打勾選項：**「在錯過指定的啟動後盡快執行工作」**。
這決定了當伺服器當機、API 關閉、或是您故意將時間設在過去時，系統的反應態度。

* **如果【不勾選】(預設)**：
  * **態度**：過去的就讓它過去，不必留戀。
  * **反應**：系統會直接拋棄所有在關機期間錯過的班車，並自動推算尋找到「未來的下一班車」，安靜地等待時間到再來執行。絕不會在您重新開啟系統時突然暴走大量補跑。
* **如果【勾選】**：
  * **態度**：使命必達，絕不漏接。
  * **反應**：一旦系統上線，它發現有過期未執行的任務，會判定為這是一筆「債務」，於是會 **立刻強制發動觸發一次** 以彌補之前的過失，然後再重新銜接回正常的預定週期。

## 5. 系統自體維護與日誌清理規則

為了確保 `quartz.db` 長期運轉下不會撐爆硬碟容量，系統內建了名為 `LogCleanupJob` 的每日全自動大掃除任務。

### 5.1 參數自訂 (appsettings.json)
您可以隨時用記事本打開 API 資料夾 (`Scheduler.Api`) 下的 `appsettings.json`，對日誌保存期限進行個別分流的修改調整：
```json
{
  "JobExecutionLogRetentionDays": 30, // 排程歷史腳本紀錄保留天數 (預設30天)
  "AuditLogRetentionDays": 365,       // 系統變更與新增稽核紀錄保留天數 (預設365天)
  "Logging": { ... }
}
```

### 5.2 執行邏輯
* **清道夫出巡**：系統會在 **每天凌晨 03:00** 於背景自動啟動這個隱藏排程。
* **空間深度回收**：尋找並抹除超過天數的過期足跡後，清道夫會向資料庫下達 `VACUUM;` 指令。這能強力重組並釋放資料庫的實體空間，將浪費的硬碟容量完整退還給作業系統。

## 6. 常見應用情境範例

### 範例 A：「資料庫備份腳本」- 每天深夜執行、不允許重疊衝突
1. **排程設定**：目標指向 `backup_db.bat`。
2. **觸發條件**：選擇「每天」，時間設為 02:00。
3. **併發規則**：選取 **「不要啟動新執行個體」**。
   *情境說明：如果昨天的資料量過大到了今天這班車的發車時間卻還沒備份完，系統會因為這個規則而選擇放棄「今天這一次觸發」，避免兩個備份程式同時讀寫同一個硬碟而導致資料卡死或覆寫失敗。*

### 範例 B：「高頻率監控腳本」- 每 5 分鐘檢查一次股市、開機立刻補跑
1. **排程設定**：目標指向 `stock_monitor.py`。
2. **觸發條件**：選擇「定期循環」，填入每 `5 分鐘` 一次。
3. **錯過機制**：**打勾**「在錯過指定的啟動後盡快執行工作」。
   *情境說明：如果伺服器不小心在 10:00 發生短暫斷線重開，直到 10:07 才上線。因為有勾選此機制，10:07 系統一開啟發現自己欠了 10:00 和 10:05 的債，便會立刻強制補派監控程式發出一筆即時檢查。*

### 範例 C：「下週單次的宣傳信件發送」- 僅執行一次
1. **排程設定**：目標指向 `send_promo_email.exe`。
2. **觸發條件**：選擇「僅一次」，日期與時間設為下週二的 09:00。
3. **錯過機制**：**不勾選**「在錯過指定的啟動後盡快執行工作」。
   *情境說明：因為下週二九點要發信，若您設定好後發現時間已經過去，即使不小心按下存檔，這套系統也會自動安靜地將該觸發條件攔阻拋棄，永遠避免您被突如其來的群聚發信嚇到。*
