# 前端使用者操作手冊

文件版本：2026-04-29  
適用對象：一般使用者、排程維運人員、交接講解人員  
相關程式：`Scheduler.Ui`

---

## 1. 前端負責什麼

前端是 Windows 桌面管理介面，負責：

- 顯示排程列表。
- 新增、編輯、刪除排程。
- 啟用、停用排程。
- 手動執行排程。
- 結束正在執行的排程。
- 查看單一排程執行記錄。
- 查看系統管理稽核記錄。

前端不直接執行排程。真正的排程由 `Scheduler.Api` 背景服務和 Quartz.NET 負責。

---

## 2. 啟動前確認

### 2.1 必須先啟動 API

UI 啟動後會連線到 API。若 API 沒開，右上角會顯示未連線。

預設 API：

```text
http://localhost:5196/
```

UI 設定檔：

```text
Scheduler.Ui/appsettings.json
```

欄位：

```json
{
  "SchedulerApi": {
    "BaseUrl": "http://localhost:5196/"
  }
}
```

### 2.2 連線狀態

主畫面右上角會顯示：

| 狀態 | 說明 |
|---|---|
| `已連線` | UI 成功連到 API |
| `未連線` | API 沒啟動、網址錯誤、Port 錯誤、網路被擋 |

## 3. 主畫面欄位說明

主畫面資料來自 `GET /api/jobs`。

| 欄位 | 說明 | 常見值 |
|---|---|---|
| 名稱 | 排程名稱 | `DailyBackup` |
| 狀態 | 排程目前狀態 | `就緒`、`執行中`、`已停用` |
| 觸發程序 | 人性化排程描述 | `每天 09:00`、`每 5 分鐘` |
| 下次執行時間 | Quartz 計算出的下一次觸發時間 | `2026/04/29 09:00:00` |
| 上次執行時間 | 最近一次手動或自動觸發時間 | `2026/04/28 18:00:00` |
| 上次執行結果 | 最近一次完成狀態 | `成功執行 (0)`、`執行失敗`、`依並發規則略過`、`已終止` |
| 作者 | 建立或編輯排程的帳號 | `DOMAIN\User` |
| 建立日期 | UI 顯示用建立時間 | 目前 ViewModel 預設值 |
| 描述 | 任務描述 | 自訂文字 |

狀態說明：

| 狀態 | 意義 |
|---|---|
| `就緒` | 排程存在，等待下次時間觸發 |
| `執行中` | 該 Job 目前有執行個體正在跑 |
| `已停用` | 排程被停用，不會自動觸發，也不能手動執行 |

## 4. 工具列按鈕

| 按鈕 | 功能 | 何時可用 |
|---|---|---|
| 新增排程 | 開啟新增排程視窗 | 永遠可用 |
| 重新整理 | 重新讀取 API 排程列表 | 永遠可用 |
| 執行 | 手動立即執行選取排程 | 選到排程且未停用 |
| 結束 | 對正在執行的排程送出中斷信號 | 選到排程且未停用 |
| 停用 | 停止排程未來觸發 | 選到排程且未停用 |

| 內容 | 編輯排程設定 | 選到排程 |
| 檢視管理記錄 | 查看新增、修改、刪除、啟停等稽核紀錄 | 永遠可用 |

- 停用狀態下，「執行」按鈕會被禁用。

## 5. 新增排程

入口：主畫面點 `新增排程`

相關程式：

- `Scheduler.Ui/AddJobWindow.xaml`
- `Scheduler.Ui/ViewModels/AddJobViewModel.cs`
- `Scheduler.Ui/ViewModels/EditTriggerViewModel.cs`
- `Scheduler.Ui/Services/SchedulerApiService.cs`

### 5.1 基本欄位

| 欄位 | 必填 | 說明 | 範例 |
|---|---|---|---|
| 任務名稱 | 是 | JobName，同群組內不可重複 | `DailyBackup` |
| 任務群組 | 否 | JobGroup，預設群組 | `DefaultGroup` |
| 描述 | 否 | 給人看的說明 | `每天備份資料庫` |
| 執行檔路徑 | 是 | 要執行的程式、批次檔或命令 | `C:\Scripts\backup.bat` |
| 引數 | 否 | 程式啟動參數 | `--mode full` |
| 工作目錄 | 否 | 程式執行的起始資料夾 | `C:\Scripts` |
| 作者 | 否 | 預設為目前 Windows 帳號 | `DOMAIN\User` |

### 5.2 執行檔路徑與工作目錄

建議：

- 若執行檔是完整路徑，工作目錄可留空或填執行所需目錄。
- 若執行檔只填檔名，請填工作目錄。

後端會做一次防呆：

```text
FileName = backup.bat
WorkingDirectory = C:\Scripts
若 C:\Scripts\backup.bat 存在，後端會改用完整路徑執行。
```

### 5.3 隱藏執行

勾選「隱藏」時：

- 不彈出視窗。
- 後端使用 `UseShellExecute=false`。
- 會擷取標準輸出 `StdOut` 與標準錯誤 `StdErr`。
- 適合背景執行 CLI、bat、python script。

不勾選「隱藏」時：

- 後端使用 `UseShellExecute=true`。
- 程式可彈出視窗。
- 無法擷取標準輸出與標準錯誤。
- 適合需要互動或需要看視窗的程式。

## 6. 觸發條件設定

入口：新增或編輯排程時，新增或編輯 Trigger。

相關程式：

- `EditTriggerWindow.xaml`
- `EditTriggerViewModel.cs`
- `TriggerDto.cs`

一個 Job 可以有多個 Trigger。

### 6.1 僅一次

用途：

- 某個固定時間只執行一次。

設定：

- 開始日期。
- 開始時間。

防呆邏輯：

- 如果時間已經過去，且沒有勾選「錯過盡快啟動」，後端不會把這個過期 trigger 交給 Quartz。
- UI 仍保留原始設定，方便日後查詢或修改。

關鍵字：

- `OneTime`
- `SimpleTrigger`
- `RepeatCount(0)`

### 6.2 定期循環

用途：

- 每 N 分鐘、每 N 小時、每 N 天持續執行。
- 例如每 5 分鐘跑一次監控。

設定：

- 開始日期與時間。
- 間隔數值。
- 間隔單位。
- 可選結束時間。

後端邏輯：

- 沒有 CronExpression，只有 RepeatInterval 時，使用 SimpleSchedule。
- 預設 `RepeatForever()`。
- 如果未勾選補跑，會重新計算下一個未來時間點，避免存檔瞬間立刻觸發。

關鍵字：

- `Interval`
- `RepeatInterval`
- `RepeatIntervalUnit`
- `SimpleSchedule`

### 6.3 每天

用途：

- 每天固定時間執行。
- 每 N 天固定時間執行。
- 固定時間後再每 N 分鐘重複一段時間。

設定：

- 每幾天。
- 每天起始時間。
- 可選重複間隔。
- 可選持續時間。

後端邏輯：

- 純每天固定時間會轉成 Cron。
- 如果加上重複間隔，後端會轉成 `DailyTimeIntervalSchedule`，因為它支援一天內從某時間開始，每隔 N 分鐘觸發。

關鍵字：

- `Daily`
- `CronExpression`
- `DailyTimeIntervalSchedule`

### 6.4 每週

用途：

- 每週特定星期執行。
- 每隔 N 週的特定星期執行。

設定：

- 星期日到星期六可複選。
- 每隔幾週。
- 起始時間。
- 可選重複間隔與持續時間。

後端邏輯：

- 星期規則會產生 Cron 的 DOW 欄位，如 `MON,WED,FRI`。
- `每隔 N 週` 是本專案自訂邏輯，Quartz 原生沒有直接支援。
- 後端會把 `WeeklyInterval` 放在 Trigger 的 JobDataMap。
- `ProcessRunnerJob` 執行時再判斷現在是否是符合 N 週倍數，不符合就直接略過。

關鍵字：

- `WeeklyInterval`
- `OnDaysOfTheWeek`
- `DOW`

### 6.5 每月

用途：

- 每月固定日期。
- 每月最後一天。
- 每月第幾個星期幾。

設定模式：

| 模式 | 範例 |
|---|---|
| 指定日期 | 每月 1、15、最後一天 |
| 指定第幾個星期幾 | 每月第三個星期五 |

後端邏輯：

- 轉成 Quartz Cron。
- 例如最後一天使用 `L`。
- 第三個星期五使用 `FRI#3`。

關鍵字：

- `Monthly`
- `L`
- `#`
- `CronExpression`

## 7. 錯過盡快啟動

UI 名稱：`在錯過指定的啟動後盡快執行工作`

後端欄位：

```text
MisfireActionFireAndProceed
```

### 7.1 不勾選

行為：

- 錯過的時間直接跳過。
- 後端會計算下一個未來時間。
- 避免 API 重啟、停用後啟用、存檔時立刻補跑。

適合：

- 備份、發信、資料匯入等不可補跑的任務。
- 時間敏感且過期就沒有意義的任務。

### 7.2 勾選

行為：

- 若真的錯過時間，Quartz 可盡快補一次。
- 本專案對 SimpleTrigger 有補跑限流，避免大量歷史觸發點瞬間爆衝。
- 但停用排程後再啟用時，本專案會從未來 trigger 重建，不補跑停用期間錯過的時間。

適合：

- 監控、同步、健康檢查等希望恢復後至少補一次的任務。

## 8. 並發規則

UI 選項：

| UI 顯示 | 後端值 | 行為 |
|---|---|---|
| 不要啟動新執行個體 | `DoNotStart` | 若同 Job 還在跑，本次略過 |
| 以平行方式執行新執行個體 | `Parallel` | 允許多個同 Job 同時跑 |
| 停止現有的執行個體 | `StopExisting` | 先中斷舊執行個體，再跑新的 |

建議：

- 資料庫備份、檔案搬移、報表產生：使用 `DoNotStart`。
- 每次執行互不影響的監控：可以使用 `Parallel`。
- 長時間卡住時必須換新任務：使用 `StopExisting`。

執行記錄：

- 若因 `DoNotStart` 被略過，會產生 `EventId=322`。
- UI 顯示為 `依並發規則略過`。

---

## 9. 最大執行時間

用途：

- 防止外部程式卡死。
- 防止一個排程永遠佔住資源。

欄位：

```text
MaxRunTimeSeconds
```

UI 可用單位：

- 分鐘
- 小時
- 天

後端邏輯：

- `ProcessRunnerJob` 會建立 `CancellationTokenSource`。
- 超過時間會呼叫 `process.Kill(true)`。
- 紀錄 `EventId=328`。

## 10. 停用與啟用

### 10.1 停用

使用時機：

- 暫時不想讓排程自動跑。
- 保留排程設定但先停止。

行為：

- API 寫入 `IsDisabled=True`。
- 呼叫 Quartz `PauseJob`。
- UI 狀態變 `已停用`。
- 停用狀態不能按 `執行`。
- API 也拒絕手動觸發。

### 10.2 啟用

使用時機：

- 恢復已停用排程。

行為：

- API 寫入 `IsDisabled=False`。
- 不直接 resume 舊 trigger。
- API 從 `OriginalTriggers` 重建新的未來 trigger。
- 避免停用期間錯過的時間在啟用後立刻補跑。

## 11. 執行記錄

入口：選取排程後點 `檢視記錄` 或開啟內容中的記錄頁。

來源：

```text
JobExecutionLogs
```

常見 EventId：

| EventId | UI 意義 | 說明 |
|---|---|---|
| `107` | 排程器已觸發工作 | 自動時間觸發 |
| `110` | 使用者已經觸發工作 | 手動執行 |
| `129` | 已建立工作處理程序 | 準備啟動外部程式 |
| `100` | 工作已開始 | Quartz Job 開始 |
| `200` | 動作已經啟動 | 外部 process 已啟動 |
| `201` | 動作已完成 | process 結束 |
| `203` | 動作失敗 | 啟動失敗或例外 |
| `322` | 並發略過 | `DoNotStart` 生效 |
| `328` | 動作已停止 | 超時、手動結束或被中斷 |

查錯建議：

- 只有 `107/110` 沒有 `200`：可能啟動前就被略過或失敗。
- 有 `200` 沒有 `201/203/328`：可能 process 還在跑，或 API 被中斷。
- `203`：看 `ErrorMessage`、`StdErr`。
- `322`：並發規則造成略過。
- `328`：手動結束、超時、或 StopExisting 中斷。

入口：主畫面 `檢視管理記錄`

來源：

```text
AuditLogs
```

| EventId | 說明 |
|---|---|
| `106` | 建立排程 |
| `140` | 更新排程 |

| `142` | 停用排程 |

用途：

- 查誰在什麼時間改了排程。
- 查排程是否曾經被停用或啟用。
- 查刪除紀錄。

## 13. 常見操作情境

### 13.1 每天凌晨備份資料庫

建議設定：

- 觸發條件：每天 02:00。
- 並發規則：不要啟動新執行個體。
- 錯過盡快啟動：不勾選。
- 最大執行時間：依備份量設定，例如 3 小時。
- 隱藏執行：通常勾選。

原因：

- 備份重疊很危險。
- 錯過備份時間後不一定應該立即補跑，避免白天佔資源。

### 13.2 每 5 分鐘監控服務

建議設定：

- 觸發條件：定期循環，每 5 分鐘。
- 並發規則：不要啟動新執行個體。
- 錯過盡快啟動：可勾選。
- 最大執行時間：小於週期，例如 3 分鐘。

原因：

- 監控恢復後補一次有價值。
- 防止上一輪卡住時一直疊加。

### 13.3 每月最後一天產報表

建議設定：

- 觸發條件：每月，指定最後一天。
- 錯過盡快啟動：視業務需求決定。
- 並發規則：不要啟動新執行個體。

原因：

- 報表通常不可重複產生。
- 若 API 關閉錯過時間，是否補產要由業務決定。

## 14. 前端查錯

### 14.1 按鈕不能按

可能原因：

- 沒有選取排程。
- 排程已停用。
- 選取項目狀態尚未刷新。

處理：

- 點選排程列。
- 按重新整理。
- 查看狀態是否為已停用。

### 14.2 執行失敗

可能原因：

- API 回傳錯誤。
- 排程已停用。
- 外部程式不存在。
- 工作目錄錯誤。
- 參數錯誤。

處理：

- 查看執行記錄。
- 看 `EventId=203` 的錯誤訊息。
- 若是停用，先啟用。

### 14.3 下次執行時間為無

可能原因：

- 單次排程已過期。
- EndAt 已過期。
- 排程沒有 live trigger，只保留原始設定。
- 排程已停用且剛被編輯後未啟用。

處理：

- 編輯 trigger，設定未來時間。
- 啟用排程。
- 檢查是否需要新增觸發條件。
