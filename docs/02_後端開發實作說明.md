# 後端開發實作說明

文件版本：2026-04-29  
適用對象：後端開發、維護人員、交接講解人員  
相關專案：`Scheduler.Api`、`Scheduler.Core`

---

## 1. 後端責任範圍

後端負責：

- 啟動 Quartz.NET Scheduler。
- 初始化 SQLite 資料庫。
- 提供 REST API 給 WPF UI 使用。
- 建立、更新、刪除 Job 與 Trigger。
- 手動觸發、中斷、停用、啟用排程。
- 執行外部程序。
- 寫入執行紀錄與管理稽核紀錄。
- 每日清除過期紀錄。

## 3. Program.cs 啟動流程

位置：`Scheduler.Api/Program.cs`

啟動順序：

1. 建立 WebApplicationBuilder。
2. 註冊 Controller、Swagger。
3. 決定 `quartz.db` 路徑。
4. 若 `quartz.db` 不存在，執行 `tables_sqlite.sql` 建立 Quartz 官方表。
5. 確保自訂表 `JobExecutionLogs`、`AuditLogs` 存在。
6. 啟用 SQLite WAL 與 MEMORY temp store。
7. 註冊 Quartz Persistent Store。
8. 註冊系統清理 Job。
9. 啟動 Quartz Hosted Service。
10. 啟動 Web API。

### 3.1 為什麼使用 SQLite

原因：

- 部署簡單，不需額外 DB Server。
- 適合單機 Windows 排程服務。
- 整包複製部署方便。

風險：

- 高併發大量寫入時仍可能遇到 DB lock。
- 所以程式啟用 WAL 並對查詢建立索引。

### 3.2 為什麼使用 WAL

程式：

```sql
PRAGMA journal_mode=WAL;
PRAGMA temp_store=MEMORY;
```

目的：

- 提高讀寫併發。
- 降低 UI 查詢 log 時阻塞排程寫入的機率。
- 降低 SQLite `unable to open database file` 或 lock 類問題。

### 3.3 Quartz Persistent Store

關鍵設定：

```csharp
q.UsePersistentStore(s =>
{
    s.PerformSchemaValidation = false;
    s.UseProperties = true;
    s.UseMicrosoftSQLite(sqlite =>
    {
        sqlite.ConnectionString = connString;
    });
    s.UseNewtonsoftJsonSerializer();
});
```

說明：

| 設定 | 原因 |
|---|---|
| `UsePersistentStore` | Job、Trigger 存入 SQLite，重啟後仍存在 |
| `PerformSchemaValidation=false` | 避免 SQLite schema 驗證過嚴造成啟動問題 |
| `UseProperties=true` | JobDataMap 以字串屬性儲存，較穩定 |
| `UseNewtonsoftJsonSerializer` | Quartz 內部序列化使用 Newtonsoft |

重要注意：

- 因 `UseProperties=true`，JobDataMap 請優先存字串。
- 讀取 bool/int 時要用 `TryParse`。
- 不要直接存複雜物件到 JobDataMap，若需要請自行 JSON 序列化成字串。

## 6. GetAllJobs 實作邏輯

方法：`JobsController.GetAllJobs()`

### 6.1 做了哪些事

1. 從 Quartz 取得所有 JobKey。
2. 取得目前正在執行的 Job。
3. 從 `JobExecutionLogs` 查每個 Job 最近結果。
4. 排除系統內部清理 Job：`System_LogCleanup`。
5. 讀取 JobDataMap 還原 JobInfo。
6. 讀取 live triggers。
7. 若有 `OriginalTriggers`，用原始設定當主資料，再合併 live trigger 的狀態與下次時間。
8. 回傳 JobInfo list。

### 6.2 為什麼要用 OriginalTriggers

問題：

- Quartz 的 Trigger 存進資料庫後，讀回來不一定能完整還原 UI 原本設定。
- 例如 UI 選了「每週 + 每隔 2 週 + 每次觸發後每 5 分鐘重複 2 小時」，Quartz 內部 trigger 不會保留 UI 頁籤與所有欄位語意。

解法：

- 建立或更新 Job 時，把 `request.Triggers` JSON 序列化後放入 JobDataMap 的 `OriginalTriggers`。
- 查詢時優先使用 `OriginalTriggers`。
- 若 live trigger 存在，就補上 `State`、`NextFireTime`。
- 若 live trigger 不存在，就顯示 `已完成/過期`，但 UI 仍能看到原始設定。

這樣可以避免：

- 單次排程過期後，UI 完全看不到原本設定。
- 使用者編輯排程時，欄位被 Quartz 轉換後失真。

### 6.3 最後執行結果如何判斷

查詢 `JobExecutionLogs`，依 `FireTimeUtc DESC, Id DESC` 排序。

EventId 對應：

| EventId | LastRunResult |
|---|---|
| `201` | `成功執行 (ExitCode)` |
| `203` | `執行失敗` |
| `322` | `依並發規則略過` |
| `328` | `已終止` |
| `100/107/110/129/200` | 過程事件，不當作結案結果 |

## 7. CreateOrUpdateJob 實作邏輯

方法：`JobsController.CreateOrUpdateJob()`

### 7.1 高階流程

```text
收到 ScheduleRequest
    ↓
檢查 Job 是否存在
    ↓
若存在，讀取原 IsDisabled 狀態，然後 DeleteJob
    ↓
建立新的 JobDetail
    ↓
將 FileName、Arguments、WorkingDirectory、Misfire、Concurrency、IsDisabled、IsHidden、Author、OriginalTriggers 寫入 JobDataMap
    ↓
依每個 TriggerDto 建立 Quartz Trigger
    ↓
套用 Misfire 與過去時間防呆
    ↓
若原本已停用，只 AddJob，不掛 live trigger
    ↓
若有 trigger，ScheduleJob
    ↓
寫 AuditLog
```

### 7.2 為什麼更新前先 DeleteJob

原因：

- Quartz 對 Job + 多 Trigger 的 replace 行為需要小心處理。
- 先刪後建能確保舊 trigger 不殘留。
- 更新時完整以目前 UI 設定為準。

防護：

- 刪除前先讀取舊 JobDataMap 的 `IsDisabled`。
- 新 Job 會保留停用狀態。

### 7.3 為什麼停用狀態只 AddJob 不 ScheduleJob

邏輯：

```csharp
if (existingDisabled)
{
    await scheduler.AddJob(job, true);
}
```

原因：

- 如果已停用的排程被編輯儲存，不能因為重建 trigger 就開始有 live trigger。
- 否則可能出現「UI 顯示已停用，但 Quartz 已經有新 trigger」的問題。
- 這是為了預防停用排程仍有機會觸發。

結果：

- Job 設定會保存。
- `OriginalTriggers` 會保存。
- 不會自動觸發。
- 使用者啟用時再由 `ResumeJob()` 重建未來 trigger。

## 8. Trigger 建立規則

Trigger 建立邏輯集中在 `Scheduler.Core/Services/TriggerFactory.cs`。Controller 只負責把 `TriggerDto` 與 misfire 設定交給 `ITriggerFactory`，建立排程與啟用排程共用同一套時間防呆邏輯，避免未來只改其中一邊。

### 8.1 CronExpression 有值

情境：

- 每天。
- 每週。
- 每月。
- 自訂 Cron。

處理：

- 若沒有 RepeatInterval，使用 `WithCronSchedule`。
- 若有 RepeatInterval 且 Cron 可解析出具體時分秒，轉成 `WithDailyTimeIntervalSchedule`。

為什麼：

- Cron 適合固定時間。
- 但「每天 09:00 開始，每 5 分鐘重複 2 小時」不是單純 Cron 容易表達的語意。
- `DailyTimeIntervalSchedule` 能表達「每日時間窗 + 間隔」。

### 8.2 CronExpression 空白且 RepeatInterval > 0

情境：

- 定期循環。
- 例如每 5 分鐘一直跑。

處理：

- 使用 `WithSimpleSchedule`。
- `RepeatForever()`。

為什麼：

- 這是 24/7 連續循環，不應被每天時間窗限制。
- SimpleTrigger 對固定間隔最直覺。

### 8.3 CronExpression 空白且 RepeatInterval = 0

情境：

- 單次執行。

處理：

- 使用 SimpleSchedule + `WithRepeatCount(0)`。
- 若 StartAt 已過去且未勾選補跑，直接 `continue`，不建立 live trigger。

為什麼：

- 防止使用者設定過去時間後，一存檔立刻補跑。
- 單次任務過期後仍可在 UI 看到原始設定，因為 `OriginalTriggers` 已保存。

## 9. Misfire 防爆設計

關鍵欄位：

```text
MisfireActionFireAndProceed
```

### 9.1 未勾選補跑

邏輯：

```text
Trigger 起始時間 <= 現在
    ↓
用 now + 2 秒當安全基準
    ↓
GetFireTimeAfter(safeBaseTime)
    ↓
把 StartAt 改成下一個未來觸發點
```

為什麼加 2 秒：

- 建立 Trigger 與交給 Quartz 中間有微小時間差。
- 如果 StartAt 剛好等於現在，很容易又被 Quartz 判定為 misfire。
- 加 2 秒可降低立即觸發或被判定過期的機率。

預防問題：

- 新增排程後立刻觸發。
- API 重啟後補跑過去時間。
- 使用者設定過去時間造成非預期執行。

### 9.2 勾選補跑

SimpleTrigger 特別處理：

```text
若 StartAt 已落後現在多個 interval
    ↓
計算 skipCounts
    ↓
把 StartAt 拉到最接近現在的上一輪
    ↓
讓 Quartz 最多感知近期的一次 misfire
```

預防問題：

- 高頻排程停機數小時後，重啟瞬間補跑大量歷史觸發。
- 外部程式被同時開啟數百次。

## 10. 停用、啟用與手動觸發防護

### 10.1 PauseJob

流程：

1. 取得 JobDetail。
2. `JobDataMap.Put("IsDisabled", "True")`。
3. `AddJob(detail, true, true)` 更新 JobDataMap。
4. `PauseJob(jobKey)`。
5. 寫入 AuditLog `142`。

為什麼要寫 `IsDisabled`：

- Quartz TriggerState 是 live trigger 狀態。
- 若排程沒有 live trigger 或 trigger 已過期，UI 仍需要知道這個 Job 是否被系統視為停用。

### 10.2 TriggerJob

流程：

1. 檢查 Job 存在。
2. 呼叫 `IsJobDisabledAsync()`。
3. 若停用，回 `409 Conflict`。
4. 未停用才 `TriggerJob`。

為什麼：

- 只靠 UI 禁用按鈕不夠。
- 外部工具或舊版 UI 仍可能直接打 API。
- API 必須是最終防線。

### 10.3 ResumeJob

舊風險：

- 直接 `scheduler.ResumeJob(jobKey)` 會讓 Quartz 根據原 trigger misfire policy 處理停用期間錯過的時間。
- 若勾選補跑，有機會一啟用就補跑停用期間錯過的任務。

目前設計：

1. 寫入 `IsDisabled=False`。
2. 呼叫 `RebuildTriggersForResumeAsync()`。
3. 先移除目前 live trigger。
4. 從 `OriginalTriggers` 還原 trigger 設定。
5. 對每個 trigger 計算 `now + 2 秒` 後的下一個未來時間。
6. 用該未來時間重建 trigger。

預防問題：

- 已停用排程被觸發。
- 停用一段時間後啟用，補跑之前錯過的排程。
- UI 狀態與 Quartz 狀態不一致。

## 11. ProcessRunnerJob 執行邏輯

位置：`Scheduler.Core/Jobs/ProcessRunnerJob.cs`

### 11.1 高階流程

```text
Quartz 觸發 Execute
    ↓
建立 correlationId
    ↓
讀取 JobDataMap
    ↓
寫 EventId 107 或 110
    ↓
檢查 WeeklyInterval
    ↓
檢查 ConcurrencyRule
    ↓
檢查 FileName
    ↓
建立 ProcessStartInfo
    ↓
依 IsHidden 決定是否 redirect stdout/stderr
    ↓
process.Start()
    ↓
寫 EventId 100、200
    ↓
等待 process 結束或中斷或超時
    ↓
寫 EventId 201、203 或 328
```

### 11.2 correlationId

每次 Execute 建立：

```text
Guid.NewGuid().ToString("N")
```

用途：

- 同一次執行會寫多筆 log。
- 用 correlationId 可以串起這些 log。

### 11.3 手動觸發與自動觸發

判斷：

```text
TriggerReason == Manual
```

EventId：

| 來源 | EventId |
|---|---|
| 自動排程 | `107` |
| 手動執行 | `110` |

### 11.4 WeeklyInterval

問題：

- Quartz Cron 原生不直覺支援「每隔 N 週」。

本專案做法：

- Trigger 仍照每週建立。
- `WeeklyInterval` 放在 Trigger JobDataMap。
- Execute 時算從 trigger 起始週到目前週過了幾週。
- 若不是 N 的倍數，直接 return。

注意：

- 不符合每隔 N 週規則時會寫入 `EventId=323`。
- UI 會顯示「依每週間隔規則略過」，方便維運辨識不是執行失敗。

### 11.5 並發規則

取得同 Job 其他執行個體：

```text
context.Scheduler.GetCurrentlyExecutingJobs()
```

規則：

| ConcurrencyRule | 行為 | EventId |
|---|---|---|
| `DoNotStart` | 有其他 instance 時略過 | `322` |
| `StopExisting` | Interrupt 舊 instance，再繼續跑新 instance | 舊 instance 通常 `328` |
| `Parallel` | 不做限制 | 正常流程 |

為什麼不用 Quartz `[DisallowConcurrentExecution]`：

- 該 attribute 只能禁止併發，不能支援 StopExisting 或 Parallel 三種模式。
- 本專案需要使用者可選擇策略。

### 11.6 隱藏與可見執行

`IsHidden=true`：

- `UseShellExecute=false`
- `CreateNoWindow=true`
- `RedirectStandardOutput=true`
- `RedirectStandardError=true`
- 可記錄 stdout/stderr。

`IsHidden=false`：

- `UseShellExecute=true`
- `CreateNoWindow=false`
- 不 redirect output。
- 讓使用者看到視窗或互動。

為什麼分開：

- .NET 限制：`UseShellExecute=true` 時不能 redirect stdout/stderr。
- 背景執行與可見互動是兩種不同需求。

### 11.7 中文輸出編碼

隱藏模式會設定：

- `Encoding.RegisterProvider(CodePagesEncodingProvider.Instance)`
- 嘗試用系統 OEM CodePage。
- 找不到時 fallback UTF-8。

目的：

- 避免繁中 Windows 執行 `ping`、`ipconfig`、bat 時 stdout 亂碼。

### 11.8 最大執行時間與中斷

邏輯：

- 建立自訂 `CancellationTokenSource` 處理 timeout。
- 與 Quartz 的 `context.CancellationToken` 合併。
- `WaitForExitAsync(linkedCts.Token)` 等待 process。
- 若取消，呼叫 `process.Kill(true)`。

EventId：

| 情境 | EventId |
|---|---|
| 正常結束 | `201` |
| process 啟動或執行例外 | `203` |
| 超時或中斷 | `328` |

## 12. LogCleanupJob

位置：`Scheduler.Core/Jobs/LogCleanupJob.cs`

### 12.1 註冊方式

在 `Program.cs` 註冊：

```text
JobKey: System_LogCleanup / System
Trigger: System_LogCleanup_Trigger / System
Cron: 0 0 3 * * ?
```

也就是每天凌晨 03:00 執行。

### 12.2 清理內容

| 表 | 設定 Key | 預設 |
|---|---|---|
| `JobExecutionLogs` | `JobExecutionLogRetentionDays` | 30 天 |
| `AuditLogs` | `AuditLogRetentionDays` | 365 天 |

清理後如果有刪除資料，會嘗試執行：

```sql
VACUUM;
```

為什麼：

- SQLite 刪除資料後，檔案大小不一定立刻縮小。
- VACUUM 可回收實體空間。

為什麼 try/catch：

- VACUUM 需要較強的資料庫鎖。
- 若排程正在高頻寫入，可能失敗。
- 這不應讓清理 Job 整體失敗，所以只記 warning。

## 15. 常見修改風險

### 15.1 改 Trigger 邏輯

請確認：

- 單次過去時間是否會立刻觸發。
- 未勾選補跑是否仍跳到未來。
- 勾選補跑是否可能大量補跑。
- DailyTimeInterval 是否跨日。
- WeeklyInterval 是否仍保留。
- OriginalTriggers 是否仍能還原 UI。

### 15.2 改停用/啟用

請確認：

- `IsDisabled` 是否正確寫入。
- 停用後手動 trigger 是否仍被 API 擋下。
- 停用狀態編輯儲存是否不會掛 live trigger。
- 啟用後是否從未來時間開始，不補跑停用期間。

### 15.3 改 ProcessRunnerJob

請確認：

- `UseShellExecute` 與 redirect 設定不能互相衝突。
- 中斷時要 kill process tree：`Kill(true)`。
- finally 一定要寫結案 EventId。
- stdout/stderr 不要造成 deadlock，必須 async read。

### 15.4 改資料庫

請確認：

- 舊版 `quartz.db` 是否能啟動。
- 新欄位用 `CREATE TABLE IF NOT EXISTS` 或安全 `ALTER TABLE`。
- 查詢是否有索引。
- 清理 Job 是否要同步更新。

## 16. 建議重構方向

目前 Trigger 建立已抽到 `TriggerFactory`，若未來要擴充，建議：

1. 持續讓 Create/Resume 共用同一套 trigger 建立邏輯，避免兩邊不同步。
2. 新增更多單元測試覆蓋：
   - OneTime 過去時間。
   - Interval 未勾補跑。
   - Interval 勾補跑。
   - Daily + RepeatDuration。
   - Weekly + WeeklyInterval。
   - Disabled edit save。
   - Resume disabled job。
3. 對 API 加 integration tests，直接用臨時 SQLite 測 Quartz behavior。
