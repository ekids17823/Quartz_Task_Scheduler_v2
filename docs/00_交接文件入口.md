## 2. 系統一句話說明

這是一套用 `Quartz.NET` 當排程核心、`ASP.NET Core Web API` 當排程引擎宿主、`WPF` 當桌面管理介面的 Windows 排程管理工具。UI 可以關閉，API 背景服務仍可繼續執行排程。

## 4. 核心資料流

```text
使用者操作 WPF UI
    ↓
MainViewModel / AddJobViewModel / EditTriggerViewModel
    ↓
SchedulerApiService 使用 HttpClient 呼叫 REST API
    ↓
Scheduler.Api / JobsController
    ↓
Quartz.NET Scheduler 建立 JobDetail + Trigger
    ↓
SQLite quartz.db 持久化排程資料
    ↓
時間到時 Quartz 執行 ProcessRunnerJob
    ↓
ProcessRunnerJob 啟動外部 exe / bat / py / cmd
    ↓
JobExecutionLogs / AuditLogs 寫入 quartz.db
    ↓
UI 輪詢 API 顯示狀態、下次執行時間、執行結果與歷程
```

## 5. 主要功能對應程式

| 功能 | 前端位置 | 後端位置 | 核心概念 |
|---|---|---|---|
| 主畫面排程列表 | `Scheduler.Ui/MainWindow.xaml`、`MainViewModel.cs` | `JobsController.GetAllJobs()` | 查詢 Job、Trigger、最後執行結果 |
| 新增排程 | `AddJobWindow.xaml`、`AddJobViewModel.cs` | `JobsController.CreateOrUpdateJob()` | 建立 JobDetail、Trigger、OriginalTriggers |
| 編輯排程 | `AddJobViewModel.cs` | `JobsController.CreateOrUpdateJob()` | 更新既有 Job，保留停用狀態 |

| 手動執行 | `MainViewModel.RunJobAsync()` | `JobsController.TriggerJob()` | 傳入 `TriggerReason=Manual` |
| 結束工作 | `MainViewModel.EndJobAsync()` | `JobsController.InterruptJob()` | Quartz Interrupt + Process Kill |
| 停用排程 | `MainViewModel.DisableJobAsync()` | `JobsController.PauseJob()` | `IsDisabled=True` + PauseJob |
| 啟用排程 | `MainViewModel.EnableJobAsync()` | `JobsController.ResumeJob()` | 重建未來 Trigger，避免補跑 |
| 刪除排程 | `MainViewModel.DeleteJobAsync()` | `JobsController.DeleteJob()` | DeleteJob |

| 自動清理 | 無 UI | `LogCleanupJob.cs`、`Program.cs` | 每日 03:00 清除過期紀錄 |

2. `Scheduler.Ui`

```powershell
dotnet run --project Scheduler.Api\Scheduler.Api.csproj
dotnet run --project Scheduler.Ui\Scheduler.Ui.csproj
```

Visual Studio 可設定多重啟動專案：`Scheduler.Api` + `Scheduler.Ui`。

## 8. 最重要的設計決策

### 8.1 UI 與 API 分離

原因：

- UI 關閉不應影響排程執行。
- 未來可支援遠端管理。
- API 可以單獨部署在排程主機上。

影響：

- UI 不直接操作 Quartz。
- 所有排程狀態以 API 回傳為準。
- 若 UI 顯示異常，先確認 API 是否可連線。

### 8.2 JobDataMap 保存設定

原因：

- Quartz 原生 Trigger 反序列化後，難以完整還原 UI 設定。
- UI 需要知道使用者原本選的是「每天、每週、每月、定期循環」等模式。

實作：

- `OriginalTriggers` 以 JSON 存在 JobDataMap。
- `MisfireActionFireAndProceed`、`ConcurrencyRule`、`IsDisabled`、`IsHidden`、`Author` 也存在 JobDataMap。

### 8.3 停用排程要有雙層保護

原因：

- Quartz 的 Pause 狀態與本系統 UI 顯示狀態要一致。
- 手動觸發 API 如果沒有檢查停用狀態，使用者仍可能觸發已停用排程。
- 啟用時如果直接 Resume，Quartz 可能依 misfire policy 補跑停用期間錯過的任務。

實作：

- `PauseJob()` 寫入 `IsDisabled=True` 並呼叫 `PauseJob`。
- `TriggerJob()` 檢查 `IsDisabled`，停用時回 `409 Conflict`。
- `ResumeJob()` 不直接 resume 舊 trigger，而是從 `OriginalTriggers` 重建未來 trigger。
- UI 的 `Run/End/Disable/Enable` command 也會依狀態禁用。

### 8.4 Misfire 不可讓系統爆衝

原因：

- API 關閉一段時間後，若存在高頻排程，重啟時不應瞬間補跑數百次。
- 使用者勾選「錯過盡快啟動」時，預期是補一次，不是補所有歷史債務。

實作：

- 未勾選補跑時，建立 Trigger 後會重新計算下一個未來時間點。
- 勾選補跑且 SimpleTrigger 落後多輪時，會把起跑點拉到接近目前時間的上一輪，降低大量補跑風險。
- 啟用停用排程時，固定從未來時間點重建 trigger，避免停用期間補跑。

## 9. 快速排查順序

### UI 顯示未連線

1. 確認 `Scheduler.Api` 是否已啟動。
2. 確認 API port 與 `Scheduler.Ui/appsettings.json` 的 `SchedulerApi:BaseUrl` 是否一致。
3. 確認防火牆或 proxy 沒有擋 localhost 或內網位址。

### 排程沒有執行

1. UI 看狀態是否為 `已停用`。
2. 看下次執行時間是否為 `無`。
3. 看 `JobExecutionLogs` 最近是否有 `107` 或 `110`。
4. 檢查 `MisfireActionFireAndProceed` 是否造成錯過時間被跳過。
5. 檢查 `EndAt` 是否已過期。
6. 檢查外部程式路徑、工作目錄、參數。

### 排程被略過

1. 看執行記錄是否有 `EventId=322`。
2. 如果有，代表並發規則 `DoNotStart` 生效。
3. 檢查上一個執行個體是否仍在跑。

### 排程啟用後不應補跑

1. 確認 `JobsController.ResumeJob()` 走 `RebuildTriggersForResumeAsync()`。
2. 確認 `OriginalTriggers` 存在。
3. 查下次執行時間應該是未來時間，不應是啟用當下立刻觸發。

## 10. 修改前檢查清單

開發者改排程邏輯前，請先確認：

- 是否會影響 `OriginalTriggers` 的相容性。
- 是否會影響 `MisfireActionFireAndProceed`。
- 是否會破壞停用排程的 `IsDisabled` 狀態。
- 是否會讓 `StartAt` 過去時間在存檔時立刻觸發。
- 是否會讓 `DailyTimeIntervalTrigger` 跨日時間失效。
- 是否會讓 `JobDataMap` 存入非字串物件。Quartz 設定 `UseProperties=true`，請優先存字串。
- 是否需要更新本 docs 文件。
