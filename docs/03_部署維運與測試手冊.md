# 部署維運與測試手冊

文件版本：2026-04-29  
適用對象：維運、開發、交接人員  
相關專案：整個 solution

---

## 1. 環境需求

| 項目 | 需求 |
|---|---|
| OS | Windows |
| .NET | API/UI/Core 使用 .NET 8；Simulator 專案目前是 .NET 10 |
| UI | WPF，需要 Windows 桌面環境 |
| DB | SQLite，檔案型資料庫 |
| 網路 | UI 要能連到 API 的 HTTP URL |

---

## 2. 開發啟動

### 2.1 建置

```powershell
dotnet build QuartzTaskScheduler.slnx -v minimal
```

### 2.2 啟動 API

```powershell
dotnet run --project Scheduler.Api\Scheduler.Api.csproj
```

預設網址：

```text
http://localhost:5196/
```

Swagger：

```text
http://localhost:5196/swagger
```

### 2.3 啟動 UI

```powershell
dotnet run --project Scheduler.Ui\Scheduler.Ui.csproj
```

或在 Visual Studio 設定多重啟動專案：

- `Scheduler.Api`
- `Scheduler.Ui`

---

## 3. 設定檔

### 3.1 API appsettings

位置：

```text
Scheduler.Api/appsettings.json
```

常用設定：

```json
{
  "Kestrel": {
    "Endpoints": {
      "Http": {
        "Url": "http://localhost:5196"
      }
    }
  },
  "JobExecutionLogRetentionDays": 30,
  "AuditLogRetentionDays": 365
}
```

### 3.2 UI appsettings

位置：

```text
Scheduler.Ui/appsettings.json
```

常用設定：

```json
{
  "SchedulerApi": {
    "BaseUrl": "http://localhost:5196/"
  }
}
```

### 3.3 改 Port

若 API 改成：

```text
http://localhost:6000
```

需要同步改：

- `Scheduler.Api/appsettings.json` 的 Kestrel URL。
- `Scheduler.Ui/appsettings.json` 的 `SchedulerApi:BaseUrl`。

## 4. 資料庫

### 4.1 quartz.db 位置

API 啟動時使用：

```csharp
Path.Combine(builder.Environment.ContentRootPath, "quartz.db")
```

開發時通常在：

```text
Scheduler.Api/quartz.db
```

或執行輸出目錄，視啟動方式而定。

### 4.2 第一次啟動

若 `quartz.db` 不存在：

1. API 會建立 SQLite 檔案。
2. 執行 `tables_sqlite.sql` 建立 Quartz 表。
3. 建立 `JobExecutionLogs`。
4. 建立 `AuditLogs`。
5. 建立索引。

### 4.3 備份

建議備份：

- `quartz.db`
- `quartz.db-wal`
- `quartz.db-shm`
- `appsettings.json`

如果使用 SQLite WAL 模式，備份時請盡量先停止 API，避免只備份到不完整檔案。

- `Scheduler.Api.exe` 或相關 DLL。

UI 發佈資料夾需要：

- `Scheduler.Ui.exe`。

- WPF UI 相關 DLL。

## 6. 建議部署方式

### 6.1 單機部署

同一台機器執行：

- API：常駐背景。
- UI：需要管理時才開。

優點：

- 最簡單。
- UI 使用 localhost 連 API。

### 6.2 遠端管理

排程主機執行 API。管理者電腦執行 UI。

需要：

- API 監聽內網 IP。
- 防火牆開 Port。
- UI `SchedulerApi:BaseUrl` 指向 API 主機。

注意：

- 目前文件未涵蓋權限驗證。若部署到多人可連環境，需評估加上 authentication。

## 7. 手動 API 測試

可用 Swagger 或 `.http` 檔。

### 7.1 查詢排程

```http
GET http://localhost:5196/api/jobs
```

### 7.2 手動觸發

```http
POST http://localhost:5196/api/jobs/DefaultGroup/DailyBackup/trigger
```

若排程已停用，預期：

```text
409 Conflict
排程已停用，不能觸發。請先啟用排程。
```

### 7.3 停用

```http
POST http://localhost:5196/api/jobs/DefaultGroup/DailyBackup/pause
```

### 7.4 啟用

```http
POST http://localhost:5196/api/jobs/DefaultGroup/DailyBackup/resume
```

## 8. 功能測試情境

### 8.1 建立單次未來排程

步驟：

1. 新增排程。
2. Trigger 選僅一次。
3. 時間設為未來 2 分鐘。
4. 儲存。

預期：

- 主畫面狀態 `就緒`。
- 下次執行時間是設定時間。
- 時間到後產生 `107/129/100/200/201` 或 `203`。

### 8.2 單次過去時間，不勾補跑

步驟：

1. 新增排程。
2. Trigger 選僅一次。
3. 時間設為過去。
4. 不勾 `錯過盡快啟動`。
5. 儲存。

預期：

- 不會立刻執行。
- 下次執行時間可能為無。
- UI 仍保留原 trigger 設定。

### 8.3 定期循環，不勾補跑

步驟：

1. 新增排程。
2. Trigger 選定期循環。
3. 每 1 分鐘。
4. 不勾補跑。
5. 儲存。

預期：

- 不應存檔瞬間立刻執行。
- 下次執行時間應是未來下一個週期。

### 8.4 停用後手動執行

步驟：

1. 選一個排程。
2. 點停用。
3. UI 嘗試按執行。
4. 或直接打 API trigger。

預期：

- UI 執行按鈕不可用。
- API 回 `409 Conflict`。
- 不會產生新的執行 log。

### 8.5 停用一段時間後啟用

步驟：

1. 建立每 1 分鐘的排程。
2. 勾選補跑。
3. 停用 3 到 5 分鐘。
4. 啟用。

預期：

- 不會補跑停用期間錯過的多次排程。
- 下次執行時間應是啟用後的未來時間。

### 8.6 並發 DoNotStart

步驟：

1. 建立每 1 分鐘排程。
2. 外部程式設定會跑超過 1 分鐘。
3. 並發規則選 `不要啟動新執行個體`。

預期：

- 第二次觸發時不啟動新 process。
- 產生 `EventId=322`。
- UI 上次執行結果顯示 `依並發規則略過`。

### 8.7 最大執行時間

步驟：

1. 建立會長時間執行的 script。
2. 設定最大執行時間 1 分鐘。
3. 執行。

預期：

- 超過 1 分鐘 process 被 kill。
- 產生 `EventId=328`。

## 9. Simulator 使用

位置：

```text
Scheduler.Simulator/Program.cs
```

用途：

- 不啟動實際排程，只推算未來觸發時間。
- 用來測每日、每週、重複間隔、持續時間、並發情境。

執行：

```powershell
dotnet run --project Scheduler.Simulator\Scheduler.Simulator.csproj
```

注意：

- 目前 Simulator 程式內參數是硬編碼，需直接改 `Program.cs` 上方設定區。
- 它是輔助工具，不會修改正式排程。

## 10. 查錯手冊

### 10.1 API 啟動失敗

檢查：

- Port 是否被佔用。
- `tables_sqlite.sql` 是否存在。
- `quartz.db` 是否被其他程式鎖住。
- .NET runtime 是否安裝。

### 10.2 UI 未連線

檢查：

- API 是否啟動。
- UI appsettings 的 BaseUrl 是否正確。
- API port 是否正確。
- 防火牆是否擋住。

### 10.3 排程不執行

檢查：

- 排程狀態是否 `已停用`。
- 下次執行時間是否為未來。
- Trigger 是否過期。
- `EndAt` 是否已過。
- 執行檔路徑是否正確。
- 工作目錄是否正確。
- 是否被 `DoNotStart` 略過。

### 10.4 執行失敗

檢查 log：

- `EventId=203`
- `ErrorMessage`
- `StdErr`
- `ExitCode`

常見原因：

- 找不到檔案。
- 沒有權限。
- 工作目錄錯誤。
- 參數錯誤。
- 外部程式本身回非 0 exit code。

### 10.5 資料庫變大

檢查：

- `JobExecutionLogRetentionDays`
- `AuditLogRetentionDays`
- `System_LogCleanup` 是否存在。
- API log 是否有 VACUUM 失敗 warning。

處理：

- 確認清理任務有執行。
- 可停 API 後手動備份與整理 SQLite。

## 11. 修改後驗證清單

每次修改後建議執行：

```powershell
dotnet build QuartzTaskScheduler.slnx -v minimal
dotnet test QuartzTaskScheduler.slnx -v minimal --no-build
git diff --check
```

若修改前端：

- 開 UI。
- 確認按鈕狀態。
- 新增、編輯、刪除排程。
- 查看 log 視窗。

若修改後端：

- 開 API Swagger。
- 測 `GET /api/jobs`。
- 測新增排程。
- 測手動 trigger。
- 測 pause/resume。
- 測 logs。

若修改排程時間邏輯：

- 用 Simulator 推算。
- 手動建立未來排程。
- 手動建立過去排程。
- 測 Misfire 勾選與未勾選。
- 測停用後啟用。

## 12. 備份與還原

### 12.1 備份

建議停止 API 後備份：

```text
quartz.db
quartz.db-wal
quartz.db-shm
Scheduler.Api/appsettings.json
Scheduler.Ui/appsettings.json
```

### 12.2 還原

步驟：

1. 停止 API。
2. 覆蓋 db 與設定檔。
3. 啟動 API。
4. 開 UI 確認排程列表。

注意：

- 若資料庫來自舊版，Program.cs 會嘗試補自訂欄位。
- Quartz 官方 schema 若大版升級，可能需要額外 migration。
